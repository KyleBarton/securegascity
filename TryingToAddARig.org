#+TITLE: Trying To Add A Rig — Incident Report
#+DATE: 2026-04-29

* Context

The finaltoon city runs under a nono sandbox that strictly isolates the
agent to the city directory. No reads or writes outside that boundary
are permitted. This is intentional: agents must not access personal
credentials, config, or history from the host user account.

We attempted to (1) add a ~crew~ agent to ~city.toml~ and (2) register
~https://github.com/gastownhall/gascity~ as a rig via ~gc rig add~.

Step (1) succeeded. Step (2) did not.

* Problems Encountered

** 1. ~git clone~ blocked on ~~/.gitconfig~

~git clone~ unconditionally reads ~~/.gitconfig~ before doing anything
else. The sandbox blocks that path, so the clone fails immediately:

#+BEGIN_SRC
fatal: unable to access '/Users/kylebarton/.gitconfig': Operation not permitted
#+END_SRC

*Workaround found:* Set ~GIT_CONFIG_NOSYSTEM=1 HOME=<city_path>~ so git
looks for a config inside the city directory instead. This got the clone
to succeed.

** 2. ~dolt config --global~ blocked on ~~/.dolt~

Setting a dolt identity (required by beads for commit authorship) writes
to ~~/.dolt/config~. The sandbox blocks that path:

#+BEGIN_SRC
Failed to load the global config: mkdir /Users/kylebarton/.dolt: file exists
#+END_SRC

(The "file exists" message is misleading — the real issue is the sandbox
denying access, not a naming collision.)

*Workaround found:* Set ~DOLT_ROOT_PATH=<city_path>/.tools/dolt~. With
that override, ~dolt config --global~ writes into the city directory and
succeeds.

** 3. ~gc rig add~ fails with "invalid managed dolt runtime state"

Even after working around problems 1 and 2, ~gc rig add~ fails:

#+BEGIN_SRC
gc rig add: bead store: invalid managed dolt runtime state
#+END_SRC

Without ~DOLT_ROOT_PATH~, it gives a different error:

#+BEGIN_SRC
gc rig add: bead store: exec beads start: dolt identity not configured
and git user.name not available
#+END_SRC

Root cause (best diagnosis): ~gc rig add~ calls ~beads start~ internally
to initialize a fresh dolt database for the new rig. However, the city
already has a managed dolt server running on port 11439 (~dolt_mode:
server~ in ~.beads/metadata.json~). The ~beads start~ call either:
- finds the running server and treats its state as invalid for a new
  rig init, or
- tries to start a second dolt instance and conflicts with the existing
  one.

The city's beads is healthy (~bd status~ shows 23 issues, server
reachable). The failure is specific to rig initialization.

* What We Know Works

- Cloning a repo into ~rigs/~ is possible with ~GIT_CONFIG_NOSYSTEM=1
  HOME=<city_path>~. The clone of ~gastownhall/gascity~ succeeded and
  is at ~rigs/gascity~.
- Setting dolt identity inside the city boundary works via
  ~DOLT_ROOT_PATH~.
- ~gc rig add --adopt~ exists for directories that already have a fully
  initialized ~.beads/~ (both ~metadata.json~ and ~config.yaml~). If we
  can initialize the rig's beads manually first, ~--adopt~ may let us
  bypass the ~beads start~ call that's failing.

* Possible Fixes

** Option A — Pre-initialize rig beads manually, then ~--adopt~

Initialize a beads database inside ~rigs/gascity~ using the city's
running dolt server (shared-server mode), then call ~gc rig add
--adopt~. This skips the internal ~beads start~ that's causing the
conflict.

The blocker: ~bd init~ from inside ~rigs/gascity~ currently detects the
city's dolt server and aborts ("workspace already initialized"). A flag
like ~--database ga~ or ~--server-port <new_port>~ might work if beads
supports isolated databases on the same server.

** Option B — Expose a dolt identity inside the sandbox

If ~gc rig add~ only fails because dolt has no identity, the right fix
may be to pre-seed a dolt config inside the city boundary before the
city starts, and ensure ~gc~ inherits ~DOLT_ROOT_PATH~ automatically
(e.g. via ~[agent.env]~ in ~city.toml~).

Add to ~city.toml~:

#+BEGIN_SRC toml
[agent.env]
CLAUDE_CONFIG_DIR = "/Users/kylebarton/Source/gasworld/finaltoon/.claude"
DOLT_ROOT_PATH    = "/Users/kylebarton/Source/gasworld/finaltoon/.tools/dolt"
#+END_SRC

Then seed the identity once (outside the sandbox) with:

#+BEGIN_SRC bash
DOLT_ROOT_PATH=.../finaltoon/.tools/dolt dolt config --global --add user.name "finaltoon"
DOLT_ROOT_PATH=.../finaltoon/.tools/dolt dolt config --global --add user.email "finaltoon@gasworld.local"
#+END_SRC

This may resolve the identity error, but the "invalid managed dolt
runtime state" error (which appears when identity IS present) suggests
there is a second, deeper conflict with the running server.

** Option C — ~gc rig add~ needs a flag for sandbox environments

The cleanest fix is a ~gc~ flag or config option that tells it:
- use the city's existing dolt server for rig beads init (shared mode), or
- skip beads init entirely and let the user run ~bd init~ manually.

This is a ~gc~ / beads feature request if no such option exists today.

** Option D — Run dolt in embedded mode for the rig

If the rig's beads is initialized with ~--backend=dolt~ in embedded
(non-server) mode, it wouldn't conflict with the city's server. Whether
~gc rig add~ supports this is unknown.
